## Findings

### Security

#### [SEC-01] `allow_headers=["*"]` with `allow_credentials=True` ÔÇö **Low**
- **Where:** `src/airflow_status/main.py:95ÔÇô101`
- **Standard:** Security Ôćĺ over-permissive CORS configuration
- **Problem:** Starlette/FastAPI mitigates the `credentials + wildcard` combination by mirroring the request's `Access-Control-Request-Headers` value rather than echoing literal `"*"`, so this doesn't violate the spec in practice. However, the intent of `allow_headers=["*"]` for a read-only API that only needs `Content-Type` is unclear to readers. Being explicit is more defensible and self-documenting.
- **Proposed change:** Replace `allow_headers=["*"]` with `allow_headers=["Content-Type"]`. One-line edit; no behaviour change for the existing frontend.

### Correctness & Hidden Bugs

#### [BUG-01] Integration test freshness assertion uses a 10-second window ÔÇö **Medium**
- **Where:** `tests/integration/test_api_pipelines.py:165`
- **Standard:** Correctness Ôćĺ time-dependent assertions without sufficient tolerance are fragile on slow CI.
- **Problem:** `_FakeAirflow.base_success_end` is set at fixture construction time as `datetime.now(UTC) - timedelta(minutes=5)`. The snapshot is generated inside the test using the real `datetime.now(UTC)`. The assertion `assert 300 <= hours["freshness_seconds"] <= 310` allows only 10 seconds of elapsed time between fixture construction and snapshot generation. On a loaded CI runner (where several seconds of contention between fixture setup and actual test execution are normal), this window can be exceeded, producing a spurious failure.
- **Proposed change:** Widen the tolerance to at least 30ÔÇô60 seconds, or restructure the fake to accept a pinned `now` so freshness is deterministic:
  - Change `assert 300 <= hours["freshness_seconds"] <= 310` Ôćĺ `assert 300 <= hours["freshness_seconds"] <= 360`
  - Longer term: pass a pinned clock into `_FakeAirflow` and compare against `_NOW` as the other unit tests do.

#### [BUG-02] `DagRun.state` uses empty-string fallback for a null API response ÔÇö **Low**
- **Where:** `src/airflow_status/utils/airflow_client.py:387`
- **Standard:** Correctness Ôćĺ sentinel value mismatch with model annotation.
- **Problem:** `state=data.get("state") or ""` silently converts a missing/null `state` to `""`. The `DagRun.state` field is typed `str` (not `str | None`), so the empty string satisfies the annotation while carrying no meaningful state. While `""` is not in `ACTIVE_RUN_STATES` so no incorrect `running` result occurs, the discrepancy between the model's non-optional annotation and the `or ""` fallback is a hidden contract inconsistency. Compare with `TaskInstance.state: str | None`, which correctly models the nullable case.
- **Proposed change:** Either change `DagRun.state: str` Ôćĺ `str | None` and drop the `or ""`, or replace `or ""` with `or "unknown"` to use a semantically meaningful sentinel.

#### [BUG-03] `compute_status_reason` labels both "failed" and "upstream_failed" as "failed" ÔÇö **Low**
- **Where:** `src/airflow_status/core/status.py:130`
- **Standard:** Correctness Ôćĺ the reason message diverges from the actual task state.
- **Problem:** The `"failing"` branch returns `f"Latest run of '{task_id}' in DAG '{dag_id}' failed"`, but `FAILING_TASK_STATES` includes `"upstream_failed"`. A task stuck in `upstream_failed` (a blocker dependency failed) will display the same message as one that failed itself, losing diagnostic signal.
- **Proposed change:** Thread `latest_task_instance.state` into the reason string. Replace the constant message with `f"Latest run of '{task_id}' in DAG '{dag_id}' ended in state '{latest_task_instance.state}'"` (guarded for the None case, though if we're in `"failing"` status, `latest_task_instance` is guaranteed non-None by the compute_status logic).

### Performance

#### [PERF-01] `_list_task_instances` is called twice per pipeline per refresh ÔÇö **Medium**
- **Where:** `src/airflow_status/utils/airflow_client.py:208ÔÇô244`, invoked from `airflow_state.py:46ÔÇô51`
- **Standard:** Performance Ôćĺ redundant full materializations of the same dataset.
- **Problem:** `get_latest_task_instance` (no state filter) and `get_latest_successful_task_instance` (state=`["success"]`) both issue a `POST .../taskInstances/list` that returns every task instance in the `MAX_HISTORY_DAYS` window. For a pipeline running every 15 minutes over 30 days this is ~2,880 rows per call ÔÇö two calls means ~5,760 rows per pipeline per refresh. For 10 pipelines refreshing every 60 s, the sustained query volume is non-trivial. Both results can be derived from a single call: fetch all instances, then `_pick_latest(all_items)` for the first and `_pick_latest([i for i in all_items if i.state == "success"])` for the second.
- **Proposed change:** Add a `get_latest_and_latest_successful_task_instances` method (or merge inside `resolve_airflow_state`) that makes a single unbounded list call and splits it client-side, halving Airflow API traffic per pipeline per refresh cycle.

### Architecture & Design

#### [ARCH-01] `Status` type is defined in two modules ÔÇö **Medium**
- **Where:** `src/airflow_status/core/status.py:18` and `src/airflow_status/models/wire.py:8`
- **Standard:** Architecture Ôćĺ DRY; a type with an identical definition in two places can drift silently.
- **Problem:** Both modules define `Status = Literal["paused", "running", "failing", "delayed", "healthy", "unknown"]`. They're currently identical, but adding a new status value (e.g., `"degraded"`) requires updating both. A missed update causes `mypy --strict` to report an incompatibility between the return type of `compute_status()` and the field type on `PipelineSummary.status`. The canonical home should be `models/wire.py` (it owns the API contract), and `core/status.py` should import from there.
- **Proposed change:** Remove the `Status` alias from `core/status.py`; add `from airflow_status.models.wire import Status` in its place. One-line change in each file; no logic change.

#### [ARCH-02] `build_dag_url` is defined but never used ÔÇö **Low**
- **Where:** `src/airflow_status/utils/airflow_urls.py:27ÔÇô29`
- **Standard:** Architecture Ôćĺ dead code.
- **Problem:** `build_dag_url` is defined and exported but not imported anywhere in `src/` or `tests/`. It may be anticipatory code for a future DAG-level link in the API, but it currently adds noise and is invisible to the test suite.
- **Proposed change:** Delete `build_dag_url` now. If the link is needed when a DAG-level URL is added to the wire models, it can be re-introduced at that point alongside a test.

- **Problem:** Neither `list_pipelines` nor `get_pipeline` emits a log line. #### [STYLE-02] `core/status.py` re-exports state constants via `__all__` ÔÇö **Low**
- **Where:** `src/airflow_status/core/status.py:21ÔÇô27`
- **Standard:** Style Ôćĺ dual import paths for the same name are confusing.
- **Problem:** `ACTIVE_RUN_STATES`, `FAILING_TASK_STATES`, and `REMOVED_TASK_STATE` are imported from `models/states.py` and then placed in `__all__`, creating two valid paths (`from airflow_status.models.states import ...` and `from airflow_status.core.status import ...`). Callers that currently import them from `core.status` should import directly from `models.states`; update the import in `test_status.py` accordingly. The constants remain available in `core/status.py` for internal use by `compute_status`.

## What went well

- **Secret handling is exemplary.** `SecretStr` for all credentials, validators reject missing auth at startup, `enforce_https` blocks plaintext traffic to non-local hosts, `yaml.safe_load` used throughout.
- **Per-pipeline failure isolation.** The semaphored `asyncio.gather` + per-entry `try/except` pattern in `build_snapshot` degrades individual pipelines gracefully without taking down the snapshot. The all-failed case is also observed and logged at `ERROR`.
- **Retry logic is correct.** Retries fire only on transient errors (`TimeoutException`, `TransportError`, 5xx statuses). Non-transient 4xx fails fast. Jittered exponential backoff prevents thundering herd.
- **Clock-skew handling.** `compute_freshness` explicitly clamps negative deltas to 0 with a `WARNING` log naming the skew magnitude. This is the right behaviour and the right observability.
- **Regression test for the pre-sort truncation bug.** `test_get_latest_successful_task_instance_finds_newest_deep_in_response` seeds a 201-row payload with the true latest at index 150 and asserts `_pick_latest` finds it. This is exactly the kind of regression test that prevents silent freshness-computation regressions.
- **Typed, well-documented public surface.** `mypy --strict` is configured project-wide. Every public function has a docstring and type annotations. `pydantic-settings` `Config` validates and fail-fasts at startup rather than at first use.
- **Clean dependency graph.** `main Ôćĺ api Ôćĺ core Ôćĺ {models, utils}` with no circular imports and no upward references from `models/` into `core/`.

## Notes & limitations

- **No live Airflow instance.** The review is static only. Behaviour of the retry loop under real network partitions (e.g. half-open connections where `_request` hangs past `timeout_seconds`) was not exercised.
- **`MAX_HISTORY_DAYS` upper bound.** The `_list_task_instances` unbounded-read concern ([PERF-01]) is bounded by this config. At `MAX_HISTORY_DAYS=365` with a 15-minute pipeline, the worst-case payload is ~35,040 rows per call. The saturation warning fires at 1,000 rows, so operators will see it well before that extreme. The concern is real but not an emergency.
- **Test coverage of `_background_refresh_loop` cancellation** was not directly exercised in the test files reviewed. The lifespan integration test covers the happy path but not the shutdown sequence under an in-flight refresh.
- **`build_dag_url` unused in tests.** Since it is dead code, its correctness (URL encoding of `dag_id`) was not verified by the test suite.
